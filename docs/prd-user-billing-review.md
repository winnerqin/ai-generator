# 用户管理与计费系统 PRD（评审版）

- 文档版本：v1.0
- 创建日期：2026-05-29
- 状态：评审中
- 适用系统：AI 创作工具后台（`ai-generator`）

## 1. 背景与目标

当前系统已具备用户登录、任务生成与基础统计能力，但缺少以下核心能力：

1. 基于角色的用户管理与菜单权限分配。
2. 用户侧账户中心（密码修改、余额、消费记录）。
3. 可配置的模型计费规则与余额扣减闭环。
4. 外部用户充值与余额不足拦截。

本期目标是完成“可运营、可追溯、可拦截”的计费与账户体系，支持后续商业化能力演进。

## 2. 范围

### 2.1 本期范围（In Scope）

1. 角色体系：系统管理员、内部用户、外部用户。
2. 角色菜单：左侧菜单按角色展示。
3. 用户中心：修改密码、查看余额、查看消费记录、充值按钮（暂不可用）。
4. 管理端用户能力：配置用户计费倍率、手动充值余额。
5. 管理端模型配置：按模型配置百万 Token 价格。
6. 任务计费：Token -> 金额，按规则进位，任务完成后扣减余额。
7. 余额校验：余额不足时阻止视频生成并提示。
8. 账务流水：充值与扣费均可追溯。

### 2.2 暂不范围（Out of Scope）

1. 在线支付/第三方支付接入。
2. 优惠券、阶梯价、套餐包。
3. 发票系统。
4. 多币种与汇率。

## 3. 角色定义

1. `system_admin`（系统管理员）
- 维护用户、角色、倍率、余额、模型价格。
- 查看账务流水与系统统计。

2. `internal_user`（内部用户）
- 使用全部业务功能（不含系统管理）。
- 可查看自身账户与消费记录。

3. `external_user`（外部用户）
- 使用对外开放业务功能（按菜单策略）。
- 可查看自身账户与消费记录。
- 余额来自管理员充值，余额不足时无法发起视频生成。

## 4. 业务规则（核心）

### 4.1 计费参数

1. 模型基准价：按“元/百万 Token”配置。
2. 用户倍率：每个用户独立配置（默认 1.0）。

### 4.2 Token 计费规则

1. 计费粒度：千 Token。
2. 最低计费：当任务 Token 小于 1000，按 1000 计费。
3. 向上进位：`2500 Token` 按 `3000 Token` 计费。
4. 公式：
- `tokens_billed = ceil(max(tokens_raw, 1000) / 1000) * 1000`
- `unit_price_yuan_per_ktoken = model_price_yuan_per_million / 1000`
- `fee_yuan = (tokens_billed / 1000) * unit_price_yuan_per_ktoken * multiplier`

### 4.3 扣费时机与原则

1. 创建任务前：执行最低费用余额校验（预拦截）。
2. 任务成功后：按实际 Token 结算扣费。
3. 任务失败/取消：不扣费。
4. 所有金额以“分”为最小存储单位，避免浮点误差。

### 4.4 余额不足处理

1. 视频生成入口提交任务时，若余额不足，返回业务错误。
2. 前端提示：`账号余额不足，请联系管理员充值。`

## 5. 功能需求

## 5.1 用户管理（管理员）

1. 创建用户时可指定角色：内部用户/外部用户。
2. 用户列表展示：用户名、角色、状态、余额、倍率、最后登录。
3. 支持修改用户倍率。
4. 支持重置密码。
5. 支持对外部用户充值（后台手工录入金额）。

## 5.2 菜单按角色分配

1. 系统管理员可见系统管理与统计菜单。
2. 内部用户可见业务全菜单（可按后续策略裁剪）。
3. 外部用户隐藏系统管理与内部专属菜单。

## 5.3 用户中心

1. 账户信息：用户名、角色、当前余额。
2. 修改密码：输入旧密码、新密码、确认密码。
3. 消费记录：按时间倒序，展示模型、Token、单价、倍率、金额、时间、任务ID。
4. 充值按钮：显示但禁用，文案“暂不可用，后续开放”。

## 5.4 模型配置（管理员）

1. 新增/编辑模型价格（元/百万 Token）。
2. 可启用/停用模型计费配置。
3. 历史任务按当时价格快照入账，不受后续改价影响。

## 5.5 计费与余额

1. 每次任务完成后自动扣费。
2. 用户余额与消费记录实时可查。
3. 充值后余额即时生效。

## 6. 数据设计（评审级）

### 6.1 用户表扩展（`users`）

新增字段建议：

1. `role_code`：`system_admin/internal_user/external_user`
2. `status`：`active/disabled`
3. `balance_cent`：账户余额（分）
4. `pricing_multiplier`：计费倍率（如 1.5000）
5. `updated_at`

### 6.2 模型价格表（`model_pricing`）

关键字段建议：

1. `model_code`（唯一）
2. `model_name`
3. `price_per_million_token_cent`
4. `enabled`
5. `created_at/updated_at`

### 6.3 账本流水表（`account_ledger`）

关键字段建议：

1. `user_id`
2. `entry_type`：`debit/credit`
3. `amount_cent`
4. `balance_before_cent`
5. `balance_after_cent`
6. `biz_type`：`omni_video/video/enhance/manual_recharge/manual_adjust`
7. `biz_id`（任务ID或业务ID）
8. `model_code`
9. `tokens_raw`
10. `tokens_billed`
11. `unit_price_cent_per_ktoken`
12. `multiplier`
13. `snapshot_json`（价格/倍率快照）
14. `operator_user_id`（管理员操作场景）
15. `created_at`

## 7. 核心流程

### 7.1 管理员配置流程

1. 配置模型价格。
2. 为用户设置倍率。
3. 为外部用户充值。

### 7.2 用户生成视频流程

1. 提交生成请求。
2. 系统按最低计费规则做余额校验。
3. 余额不足：拒绝并提示。
4. 余额充足：创建任务。
5. 任务成功回写 Token 后，按规则结算扣费并写账本。
6. 用户中心可立即看到余额变化和消费明细。

## 9. 非功能与风控要求

1. 并发安全：扣费与充值必须事务化。
2. 可追溯性：任何余额变化必须写流水。
3. 精度一致：统一使用分（整数）计算与存储。
4. 审计能力：管理员可按用户/时间查询资金流水。
5. 兼容性：MySQL 8 可运行。

## 10. 验收标准（UAT）

1. 可创建内部/外部用户，并正确显示角色菜单。
2. 管理员可设置模型价格与用户倍率。
3. 管理员可为外部用户充值，余额即时更新。
4. 生成任务成功后发生正确扣费，账本可查。
5. `token<1000` 按 `1000` 计费；`2500` 按 `3000` 计费。
6. 余额不足时视频生成被拦截并展示明确提示。
7. 用户中心可修改密码、查看余额、查看消费记录。
8. 充值按钮可见且不可点击，标记“暂不可用”。

## 11. 里程碑建议

1. M1（基础）：角色字段、权限装饰器、菜单分配。
2. M2（计费）：模型价格、倍率、扣费引擎、账本。
3. M3（用户侧）：用户中心、消费记录、余额不足提示。
4. M4（运营）：管理端流水查询、统计补充、联调验收。

## 12. 待确认项（评审会决策）

1. 外部用户可见的具体菜单清单（是否包含部分工具页）。
2. 扣费触发时点：仅成功扣费，还是提交即冻结、完成后结算。
3. 金额舍入策略（四舍五入/银行家舍入）。
4. 是否对图片任务同步纳入余额扣费（本期默认先覆盖视频链路）。

## 附：示例计费

- 模型价格：`120 元/百万 Token`
- 用户倍率：`1.5`
- 原始消耗：`2500 Token`

计算：

1. `tokens_billed = 3000`
2. `unit_price = 120/1000 = 0.12 元/千Token`
3. `fee = 3 * 0.12 * 1.5 = 0.54 元`

最终扣费：`54 分`
